REQ-001 需求基线

本篇建立「纵列双发矢量推力飞行器」仿真项目的需求基线：编号规则、属性定义、功能需求、性能约束、接口需求、验证方法映射与当前满足状态。需求状态以当前代码与测试为证据，不成熟度见 CFG-000 与 VER-001。

1 需求工程方法

1.1 编号规则

需求编号格式为：

REQ-<域>-<三位序号>
域 代码 覆盖内容
仿真动力学 SIM 六自由度积分、子步调度、地面约束
控制 CTL 通道映射、SAS 增稳、执行机构
推进 PRP 推力、反扭、差速分配、电机滞后
气动 AER 线性气动模型、空速下限
参数治理 PAR 单一事实源、漂移检测
测试验证 TST 单元测试、回归基线
用户界面 UI 滑块、HUD、演示、开关
性能约束 PERF 实时性、无构建、零依赖
接口 IFC JSON schema、生成契约、DOM 契约

1.2 需求属性

每条需求至少包含以下属性：

  • 标题：需求一句话描述；
  • 来源：该需求来自何处（如 CFG-000、CONOPS-001、内部设计决策）；
  • 优先级：高 / 中 / 低；
  • 验证方法：检查（I）、分析（A）、测试（T）、演示（D）或其组合，映射到 VER-001；
  • 状态：已满足 / 部分满足 / 未满足；
  • 证据：指向源码模块、测试文件或仿真页面元素。

1.3 验证方法映射

代码 方法 含义 适用场景
I 检查 Inspection 人工审查文档 / 代码 / 配置，确认实现存在且符合约定 参数治理、接口契约、代码结构
A 分析 Analysis 通过数学推导、静态检查或代码走读确认属性 守恒量、稳定性、数值边界
T 测试 Test 通过 node --test 自动化测试执行并断言 单元功能、回归基线
D 演示 Demonstration 通过 Web 仿真页面操作并观察结果 UI 输入、演示序列、HUD

5 非目标

以下工作明确不在 C0 基线范围内，后续若需纳入，应通过 GOV-002 变更流程并新增需求：

  • 全包线飞行验证或飞行品质等级评定；
  • 硬件在环（HIL）仿真、传感器模型、嵌入式飞控代码；
  • CFD、风洞或涡格法高保真气动数据库；
  • 失速 / 深失速 / 尾旋等非线性气动建模；
  • 自动导航、航迹规划、任务管理系统。
