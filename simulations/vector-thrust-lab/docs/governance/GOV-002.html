GOV-002 信息与配置管理

本篇规定项目的源信息格式、文档编号规则、参数治理工作流、配置管理与变更控制要求。本篇回答「信息以什么形式存在、如何追溯、如何变更」，不展开具体物理推导或控制律设计。

1. 信息模型

项目采用三类信息载体，职责分离如下：

载体 格式 用途 生成物示例
文档源 Markdown 人读的技术文档、决策、参考信息 docs/**/*.md
参数源 JSON 单一事实源，含来源/置信度/单位/描述 models/aircraft-model.json
生成物 JS/ESM/HTML 由源信息生成，禁止手工编辑 parameters.mjs、docs/*.html

生成物头部必须标注单一事实源与生成工具。例如 parameters.mjs 顶部包含：

//  单一事实源：仓库根目录 models/aircraft-model.json
//  本文件由 tools/sync-params.py 生成 —— 请勿手工编辑

仿真子系统 simulations/vector-thrust-lab/docs/ 下的 HTML 文件属于构建产物，未来由 tools/build-docs.py 从 Markdown 源生成；当前为早期手工版本，状态见 §4。

3. 参数治理工作流

3.1 参数追溯链

flowchart LR
    accTitle 参数单一事实源追溯链
    accDescr 展示 aircraft-model.json 如何通过 sync-params.py 生成 parameters.mjs，并流向 core 模块与测试基线；--check 形成漂移检测回路。

    JSON["models/aircraft-model.json\n52 参数 + 来源/置信度"]
    TOOL["tools/sync-params.py\n生成 + --check 校验"]
    PMS["simulations/.../parameters.mjs\n勿手改"]
    CORE["src/core/*\nmath/control/..."]
    TESTS["tests/*.test.mjs\n单测断言"]
    BASE["tests/fixtures/\nregression-baseline.json"]
    CI["CI / 本地提交前\npy -3.12 tools/sync-params.py --check"]

    JSON -->|生成| TOOL
    TOOL -->|写入| PMS
    PMS -->|导入| CORE
    CORE -->|驱动| TESTS
    CORE -->|驱动| BASE
    CI -->|比对| TOOL
    TOOL -.->|漂移告警| JSON

3.2 变更步骤

  1. 在 models/aircraft-model.json 中修改参数值/来源/置信度；
  2. 运行 python tools/sync-params.py 重新生成 parameters.mjs；
  3. 运行 node --test tests/*.test.mjs 与回归测试；
  4. 若参数变更导致基线漂移，按 §6 行为保持红线流程重采基线；
  5. 提交时一并包含 JSON、mjs、测试基线（如更新）与文档说明。

3.3 参数状态分类

状态 含义 升级条件
MODEL-DEFAULT 模型默认值，无外部证据 初始状态
ASSUMED 基于经验或粗略假设 需记录假设来源与理由
ESTIMATED 经分析/类比计算得到 需保留计算过程与输入
DATASHEET 来自厂商/公开数据手册 需记录数据手册编号/链接/版本
BENCH-TESTED 经地面台架测试标定 需将试验记录写入 registers/参数数据手册
FLIGHT-IDENTIFIED 经飞行试验辨识 需将飞行数据与辨识方法写入 registers/参数数据手册

降级（如 FLIGHT-IDENTIFIED → ESTIMATED）仅在发现原证据存在重大缺陷时执行，需在 registers/参数数据手册 中说明原因。

4.2 行尾与编码

  • 文本文件默认 LF 行尾，由 .gitattributes 自动处理：
* text=auto eol=lf
  • Windows 批处理文件（.bat、.cmd）强制 CRLF；
  • 二进制资产（图片、字体、CAD 文件等）在 .gitattributes 中显式标注 binary；
  • 所有源代码与文档采用 UTF-8 无 BOM 编码。

5. 变更管理

5.1 行为保持红线清单

下列数值/算法契约属于行为保持红线，修改会改变仿真可观测行为。任何有意变更必须：

  1. 更新本清单或相关文档中的对应描述；
  2. 重采回归基线 tests/fixtures/regression-baseline.json；
  3. 在提交说明中明确记录变更原因与影响。
红线项 当前值 所在位置
帧 delta 上限 0.05 s frameCap
积分子步上限 0.004 s maxStep
电机一阶滞后时间常数 0.28 s tauM
差速开方分配 ωf = ω0·sqrt(1+Δω)、ωt = ω0·sqrt(1−Δω) propulsion.mjs
SAS 摆角限幅 ±dMax = ±0.4363 rad（25°） dMax
SAS 差速限幅 ±0.7 dwMax
积分限幅 俯仰 intTh ±0.5 rad、滚转 intPhi ±0.3 rad intThMax、intPhiMax
UI 差速幅值 ±0.55 dwUiMax
空速下限 0.5 m/s vMin
地面约束 pos.z > 6.2 且 vWorld.z > 0 groundZ + state.mjs
四元数积分 显式欧拉右乘 + 每步归一化 math.mjs
自动综合演示 900 ms 触发 demo.mjs
完整复位行为 热配平复位并恢复可重复动态状态 state.mjs
欧拉角提取 theta = −asin(R13) 约定 math.mjs、tests/math.test.mjs
渲染/物理分离 渲染几何电机位置 ±1.78 m ≠ 物理力臂 0.62 m aircraft-view.mjs vs aircraft-model.json

5.2 文档变更流程

  1. 修改 Markdown 源文件；
  2. 若文档变更影响交叉引用，同步更新引用处（引用格式：「见 CTL-001 §3」）；
  3. 若变更涉及行为红线，执行 §5.1 全部步骤；
  4. 本地运行 node --test tests/*.test.mjs 确认无回归失败；
  5. 按阶段提交，提交信息中文，尾部加 Co-Authored-By 行。

5.3 构建产物管理

  • simulations/vector-thrust-lab/docs/*.html 为构建产物，未来由 tools/build-docs.py 从 Markdown 生成；
  • 当前阶段产物可随源码提交，但应明确标注其「生成物」属性；
  • 禁止在构建产物中直接做持久化修改，所有修改应在 Markdown 源中完成并重新构建。
