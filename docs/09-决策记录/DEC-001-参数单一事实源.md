# DEC-001 参数单一事实源

## 状态

已接受

## 日期

2026-07-20

## 背景

在重构之前，仿真项目的物理参数以硬编码字面量形式分散于 `app.js` 多处：

- 推力系数、惯量、气动导数、SAS 增益、数值阈值等重复出现；
- 同一名称的参数在不同函数中可能取值不一致，导致调试时难以判断「当前生效值」；
- 参数没有来源、单位、置信度记录，无法区分「模型默认值」「台架标定值」与「飞行辨识值」；
- 新增或修改参数时需要手工同步多处，容易遗漏并引入回归失败。

随着 6-DOF 仿真从单体脚本演进为分模块工程（core 纯计算层 + browser 适配层），上述问题成为可控性与可追溯性的主要障碍。

## 决策

采用 **JSON 单一事实源 + 代码生成 + 漂移检测** 的参数治理方案：

1. **单一事实源**：所有物理/控制/数值参数统一存放于 `models/aircraft-model.json`，每个参数包含 `name`、`value`、`unit`、`description`、`source`、`confidence` 字段；
2. **代码生成**：由 `tools/sync-params.py` 读取 JSON 并生成 `simulations/vector-thrust-lab/src/core/parameters.mjs`，生成物头部标注「单一事实源」与「请勿手工编辑」；
3. **漂移检测**：`tools/sync-params.py --check` 比对 JSON 与生成物内容，若不一致则返回非零退出码并在 stderr 输出漂移告警；
4. **可追溯**：`source` 字段使用 6 级状态（MODEL-DEFAULT / ASSUMED / ESTIMATED / DATASHEET / BENCH-TESTED / FLIGHT-IDENTIFIED），`confidence` 字段记录置信度，支持参数成熟度演进追踪。

## 备选方案

| 方案 | 说明 | 否决理由 |
|------|------|----------|
| 保持硬编码字面量 | 参数继续分散在 `app.js` 或各模块中 | 多处复制必然漂移，无法追溯来源与置信度 |
| 独立 JS 常量手工维护 | 在 `parameters.mjs` 中直接写 `export const P = {...}` | 形成 JSON 文档与 JS 常量双源，无法避免漂移 |
| 构建期打包注入 | 使用 Vite/Rollup 等工具在构建时注入参数 | 引入 bundler，违背「无构建原生 ESM」约束，增加工具链复杂度 |

## 后果

### 正面影响

- **单点修改**：修改参数只需编辑 `models/aircraft-model.json` 并运行生成工具；
- **来源可信**：每个参数的来源状态与置信度显性记录，便于后续台架或飞行标定后升级；
- **CI 兜底**：`--check` 可在提交前或 CI 中自动检测「改了 JSON 却忘了同步生成物」的疏漏；
- **文档与代码同源**：参数数据手册、模型规范、代码实现共享同一 JSON 源。

### 代价与约束

- 改参数后必须运行 `python tools/sync-params.py`（或 `py -3.12 tools/sync-params.py`），否则生成物与源不一致；
- 参数变更若影响行为红线，需同步重采回归基线并在提交说明中记录原因；
- 当前全部参数状态为 `MODEL-DEFAULT`，模型仍处于概念级，不可因治理改进而误判为高保真。
